一页结论（给管理层）

  • 项目已具备“可交付”的主链路：AI 语义生成 → 编译 → SVG/PPTX 导出 → QA，且内部批处理路径已有状态机和工件契约。

技术路线建议（30/60/90天）

阶段A（第1-2周）：先做“对齐”，不做大改

  • 统一 v2-render 的产品定义： 方案1：保留 AI 强依赖，同时改 README/CLI 文档； 方案2：提供 --skip-ai-review，恢复离线渲染能力。
  • 把这条契约加进 test_doc_drift.py，防止再次漂移。

阶段B（第3-6周）：工程质量收敛

  • 先清理 mypy 的 37 条类型错误（优先 template_manifest.py / html_parser.py）。
  • 把 ruff 从“全量修”拆成“增量门禁”：新改动必须过，历史问题分批清。
  • 对 v2/io.py 默认输出策略加 run-id 或时间戳，避免覆盖。

阶段C（第7-10周）：职责解耦与能力增强

  • 收敛 bridge 入口，把 _write_svg_project 与导出调用统一到 batch/pptmaster_bridge.py（V2 调用桥接层，不再直接持有外部脚本细节）。
  • 补强 post-export QA：增加版式稳定性、关键文本溢出等规则检查。
  • 把 source_refs 从“整页共享”升级到“块级/要点级”。

阶段D（第11-13周）：产品化稳定阶段

  • 形成“命令契约清单 + 回归清单 + 发布清单”三件套，接入 release gate。
  • 沉淀可观测指标：失败率、重试次数、平均生成时长、degraded 占比。
