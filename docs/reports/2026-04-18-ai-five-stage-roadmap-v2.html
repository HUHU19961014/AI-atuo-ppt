Enterprise-AI-PPT 五阶段 AI 改造路线（V2）

文档日期：2026-04-18 ｜ 目标：把“需求澄清/大纲/语义Deck/质量重写/ReviewPatch”统一成 pptmaster 风格的 AI 主导流水线。

核心结论
旧任务清单需要重写，且要按阶段重排优先级。
方法
AI 负责生成与评审，工程负责契约、验证、回滚。
运行模式
默认 Agent-First（免 key），Runtime-API 仅作兼容模式。

为什么要改写旧清单

  • 旧清单工程债务占比过高（mypy/ruff 等），与“五阶段 AI 对齐”目标不匹配。
  • `v2-*` 与 `batch-make` 的阶段语义不一致，继续按旧清单推进会重复建设。
  • 用户当前关心点是“什么时候走 Agent、什么时候才需要 API key、怎么稳定产出”，需要任务直接回答这些问题。

任务清单（V2）

Task A01：五阶段统一契约与工件模型P0

目标：统一 `v2-*` 和 `batch-make` 阶段定义、输入输出 schema、工件命名。

Why：不统一契约，AI 越多，漂移越快。

How：盘点命令 -> 建立标准阶段表 -> 更新 `docs/CLI_CONTRACTS.json` -> 加入 drift 测试。

When：2026-04-19

注意事项：Review 与 Rewrite 禁止混名；所有别名必须统一映射。

Task A02：需求澄清阶段 AI 化对齐P0

目标：从“topic 直入 outline”升级为“Clarify -> Outline”。

Why：先定策略再开画布，才是 pptmaster 风格。

How：抽公共 Clarify 服务 -> `batch/preprocess.py` 接入 -> 输出 `clarify_result.json`。

When：2026-04-20 ~ 2026-04-21

注意事项：限制自由文本输出；缺字段可降级，不能整链路崩。

Task A03：大纲阶段策略化P0

目标：增强 Outline 的叙事节奏、页数边界、结构检查。

Why：结构错，后面全部返工。

How：扩 schema + prompt -> 增加大纲质量检查 -> validation feedback 重试。

When：2026-04-22

注意事项：标题可读性必须规则化校验；不要越界做视觉细节。

Task A04：语义 Deck 候选化 + 可追溯化P0

目标：默认多候选生成并自动选优，提升稳定性。

Why：AI 单次输出波动大，多候选更稳。

How：接入 batch candidate 评分 -> 扩展块级 `source_refs` -> 增加回归样例。

When：2026-04-23 ~ 2026-04-24

注意事项：先 2-3 候选控制成本；评分不能只看格式，要看证据覆盖。

Task A05：质量重写阶段分层化P0

目标：从单次 rewrite 变为有限回路（规则判定 + 重写 + 分流）。

Why：很多问题可修，不该一次失败直接中断。

How：拆 pre-export QA 与 rewrite executor -> 增加轮次上限 -> 标准化 qa 报告。

When：2026-04-25 ~ 2026-04-27

注意事项：每轮必须回归，防止“修A坏B”。

Task A06：ReviewPatch 阶段统一化P0

目标：把 `v2-render` review patch 下沉为公共阶段，`v2` + `batch` 共用。

Why：不同链路质量标准必须一致。

How：抽 `visual_review` 公共 API -> batch 导出后接入 -> patch 冲突保护。

When：2026-04-28 ~ 2026-04-30

注意事项：patch 必须严格字段校验；只自动修 blocker。

Task A07：Agent 模式与 API 模式契约化P0

目标：明确默认 Agent-First 免 key，Runtime-API 才要求 key。

Why：“用了 agent 还被要求 key”是当前最大认知冲突。

How：更新 `CLI_CONTRACTS`（`ai_execution_mode`/`requires_api_key`）+ CLI 模式开关 + 文档与契约测试。

When：2026-05-01

注意事项：Agent 模式与 Runtime-API 模式必须可区分、可观测、可回归。

Task A08：回归基线与指标P1

目标：固定样本回放 + 成本/质量/时延指标。

Why：没有基线就无法判断改造是否成功。

How：建立 baseline fixtures -> 批量回放脚本 -> 门禁阈值。

When：2026-05-02 ~ 2026-05-05

注意事项：模型版本必须入库，否则数据不可比。

Task A09：灰度上线与回滚手册P1

目标：新五阶段逐步切流量，确保可回退。

Why：生产链路不可一次性全量切换。

How：feature flag -> 10% 灰度 -> 50% 灰度 -> 全量。

When：2026-05-06 ~ 2026-05-08

注意事项：回退必须配置化，不允许人工热修顶替。

Task A10：存量工程债务收口P2

目标：五阶段稳定后再做 mypy/ruff 收口。

Why：先业务链路、后工程洁癖，符合当前收益。

How：核心路径先清理 -> 增量门禁 -> 分批回归。

When：2026-05-09 ~ 2026-05-12

注意事项：禁止用 `Any` 粗暴消错；风格修复不改语义。

结论

是的，必须重写并调整任务。 新版不是推翻旧计划，而是把旧计划中高价值项吸收到“五阶段 AI 对齐”的主线里，并默认采用 `pptmaster` 风格的 Agent-First（免 key）方式。
